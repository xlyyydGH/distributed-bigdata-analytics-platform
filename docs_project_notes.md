# 项目难点与收获

## 项目名称

分布式大数据处理与分析平台  |  课程/团队项目  |  Spark / Kubernetes / Linux / MySQL  |  2022.05 - 2024.05

## 简历描述

项目描述：基于大数据与容器化技术构建数据处理与分析平台，实现用户行为数据的采集、清洗、离线计算、指标存储、API 查询与分析报表展示。

- 基于 Spark 设计离线批处理流程，对用户行为日志进行清洗、宽表关联和多维聚合，产出 DAU、CTR、CVR、GMV、品类销售、Top 商品和用户留存等指标。
- 设计原始层与指标层数据结构，将原始日志存储于 `data/raw`，将聚合结果沉淀到 `warehouse/marts`，并支持写入 MySQL/SQLite 供 API 和报表查询。
- 基于 Kubernetes 设计 MySQL、指标 API 与 Spark Job 的容器化部署方案，配置资源 requests/limits，支持服务统一调度和运行环境一致性管理。
- 提供本地 Pandas 兜底链路，在无 Spark 环境下仍可跑通端到端 Demo，便于调试、演示和结果校验。

## 项目难点

1. **批处理性能与指标口径一致性**

   用户行为日志包含曝光、点击、加购、购买等多类事件，指标计算涉及去重、漏斗转化、GMV、品类聚合、Top 商品和留存分析。难点在于既要保证计算链路可扩展，又要保证同一指标在 CSV、数据库、API 和报表中的口径一致。

2. **大规模数据下的 Join 与 Shuffle 控制**

   Spark 计算中，行为日志与商品维表关联、按日期/品类/用户聚合都会触发 Shuffle。需要关注分区数、宽表关联方式、聚合粒度和输出文件数量，避免小数据能跑、大数据性能急剧下降的问题。

3. **离线任务与在线查询的解耦**

   平台将离线 ETL 结果沉淀为指标表，再由 API/报表读取，避免在线接口直接扫描原始日志。这个设计降低了查询延迟，也让批处理任务、数据库和服务层职责更清晰。

4. **容器化部署与资源调度**

   Kubernetes 部署不仅是把服务放进容器，还要考虑 MySQL、API、Spark Job 的启动顺序、资源 requests/limits、Service 暴露、Job 失败重试和运行环境一致性。

## 专业收获

- 系统理解了大数据平台从原始数据接入、离线计算、指标建模、数据落库到 API/报表服务的完整工程链路。
- 掌握了 Spark 批处理中的核心概念，包括 DataFrame API、宽表 Join、多维聚合、Shuffle、分区和批处理输出管理。
- 理解了指标平台设计中“计算口径统一”和“离线结果服务化”的重要性，能够把业务指标拆解为可维护的数据表和接口。
- 熟悉了 Kubernetes 在数据平台中的资源编排价值，包括服务部署、任务调度、资源限制和环境一致性管理。
- 提升了 Linux/容器化环境下排查问题的能力，包括路径挂载、依赖安装、日志定位和任务失败恢复。
